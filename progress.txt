## Codebase Patterns

- `ownerUserId` on skill table is now nullable (system-default vault skills have no user owner)
- output zod schema for skill uses `ownerUserId: z.string().nullable()`
- all skill read queries filter by `inArray(skill.ownerVaultId, enabledVaultIds)` — never by `ownerUserId`
- all skill/graph outputs include a `vault` object (id, slug, name, type, color, isReadOnly) for UI/CLI to display vault context
- single-item fetches (getById/getBySlug) use `assertCanRead` from vault-access for membership check
- list/search/graph use `getEnabledVaultIds` for bulk filtering (disabled vaults excluded)
- schema files live in `packages/db/src/schema/`, barrel-exported from `index.ts`
- tables use uuid PKs with `defaultRandom()` (except auth tables which use text PKs)
- snake_case in DB, camelCase in JS; timestamps always `created_at` + `updated_at`
- relations defined in same file as tables, named relation keys for disambiguation
- circular imports between schema files are fine (drizzle relations are lazy)
- auth `userRelations` in `auth.ts` references tables from other schema files
- enterprise vault management routes use a shared owner/admin guard (`assertCanManageEnterpriseVault`) and normalized email matching for invitations
- destructive one-off data migrations in auth use `--apply` opt-in; default run is dry-run and deletion is blocked when unresolved mention targets remain
- CLI identifier resolution should go through `resolveSkillIdentifier` to keep UUID/plain-slug/`<vault>/<slug>` handling consistent across commands
- CLI sync install keys must use `buildVaultScopedInstallKey(vault.slug, skill.slug)` and stale pruning must compare both expected keys and server skill IDs (handles same-slug collisions + disabled vault removal + legacy lock key cleanup)
- CLI vault selection flags should accept either UUID or vault slug; resolve slugs through `vaults.listMine` before write mutations
- browser auth for UI verification:
  - run `browser-use profile list-local`
  - open app with a real profile: `browser-use --browser real --profile "<profile>" open http://localhost:3001/login?next=%2Fsettings`
  - complete login in that same browser context, keep it open, then run checks on `/settings`
  - if profile auth is flaky, set `better-auth.session_token` via `browser-use cookies set` as a fallback
  - local fallback token file: `.env.ralph.local` (`BETTER_AUTH_SESSION_TOKEN`), then run `source .env.ralph.local && browser-use --session ralph-vault cookies set better-auth.session_token "$BETTER_AUTH_SESSION_TOKEN" --domain localhost --path / --secure --http-only --same-site None`
